## Q1：自我介绍

**候选人：** B 端工作台（2022 - 2024）：负责医药 B 端平台核心模块开发，主导老项目向 Monorepo 微前端架构的渐进式迁移，搭建 iframe 微前端方案实现子应用独立部署与运行时隔离，沉淀业务组件库降低团队协作成本。AI 低代码编程平台（2024 - 2025）：独立负责 AI 低代码编程平台全栈开发，构建 Design-to-Code、服务端在线编译、CLI/MCP/Skill 工具链等核心能力，打通本地 IDE 到平台的开发-发布链路。医药 AI 产品开发（2025 - 至今）：参与医药健康 AI 智能产品从 0 到 1 搭建，主导 B 端 AI 助手从 Workflow 到 Agent 架构演进，自研小程序 SSE 流式引擎，搭建 Node.js BFF 中间层支撑流式传输与安全审核。

最有成就感的事是，在负责医药 B 端平台核心模块的核心开发与架构优化，涵盖数据看板、营销工具、补贴管理、门店圈选等多条业务线；同时主导老项目向 Monorepo 微前端架构的渐进式升级。1. 微前端架构升级：主导 B 端老项目向 Monorepo 微前端架构的渐进式迁移，搭建子项目 Lint 规范、打包配置与 CI/CD 发布流水线。2. 搭建 iframe 微前端架构：基于 iframe 封装轻量级微前端工具，支持主子页面路由感知和同步，实现子应用独立开发、独立部署与运行时隔离。3. 业务组件库建设：沉淀医药 B 端业务组件库，将数据看板、表单配置、权限控制等高频场景抽象为通用组件，作为 Monorepo 独立 package 在多个子应用间共享复用，配合组件文档降低团队接入成本。

## Q2：微前端选型

**面试官：** 你选择了 iframe 方案来做微前端，这个选型背后的考量是什么？当时有没有调研过 qiankun、micro-app、Module Federation 这些方案？最终为什么选了 iframe 而不是它们？

**候选人：** qiankun 的改造工作量比较大，虽然比较完善，但我们需要一个渐进式的框架。micro-app 是一个渐进式的框架，在我们改造时还不够完善，当时功能还不是很完整。Module Federation 适合做方法的共享，而我们需要的是页面级别的共享。

---

## Q3：iframe 方案的痛点与路由同步

**面试官：** iframe 方案天然存在一些痛点，比如主子应用之间的通信、样式隔离虽然天然具备但同时也带来了体验割裂的问题、还有白屏加载性能等。你在实际落地过程中，遇到的最棘手的问题是什么？你是怎么解决的？比如你提到了"主子页面路由感知和同步"，这个具体是怎么实现的？主应用前进后退的时候，iframe 内部的路由状态是怎么保持同步的？

**候选人：** 其他的体验问题确实存在，我们主要解决了主子页面路由感知和同步问题。在 iframe 中，子页面路由变化会通过 postMessage 传递到父路由，再在父路由中硬编码进 URL query 来实现的路由感知。

---

## Q4：路由恢复与体验优化

**面试官：** 第一，用户直接刷新页面的时候，你怎么保证 iframe 内部能恢复到正确的子路由状态？也就是说，你的 query 参数是怎么在初始化时回传给子应用的？第二，你提到了其他体验问题"确实存在"，那在你们的业务场景下，这些问题有没有对用户体验产生实际影响？你们有没有做过什么优化，还是说在你们的场景下这些问题可以接受？

**候选人：** 1. 在载入子页面时，有一套解析机制，会解析在 URL query 里拼接的特定 Key，再将拼接好的路由参数给 iframe 加载。2. 其他问题是页面保活，白屏这些吧。由于子页面的量级都不高，确实会有一定的加载延迟，但由于我们场景对性能体验没有太高要求，所以目前还可以接受。

## Q5：小程序 SSE 流式引擎

**面试官：** 你提到基于 `wx.request` 的 `enableChunked` 来实现 SSE Client，能具体讲讲你这个引擎的核心设计吗？比如"二进制切帧"和"跨 chunk 半帧合并"是怎么回事？为什么会出现半帧的情况？你是怎么处理的？

**候选人：** 二进制切帧主要是为了解决小程序部分 iOS 的 JavaScriptCore 对于 TextDecoder 兼容性不足的问题，我们通过使用自己实现的 Utf8toStr 方法，实现了小程序低版本基础库的兼容。跨 chunk 半帧合并是指，SSE 有时候会在每个 chunk 里，不返回一个完整的 JSON，会通过多个 chunk 来返回，所以需要通过寻找 data 的结束标识来合并 chunk 的内容。

---

## Q6：双重超时守护

**面试官：** 你提到了"首帧 90s + 帧间 5s 双重超时守护"，这个设计的考量是什么？为什么首帧和帧间要用不同的超时时间？另外，当超时触发的时候，你的引擎是怎么处理的？是直接断开重连，还是有其他的容错策略？

**候选人：** 首帧超时保护的是"业务层等待"（模型推理慢，允许长），帧间超时保护的是"连接健康度"（流已经在跑但突然卡住，必须短），两者对应的失败模式不同，所以时长差了近 20 倍。超时触发时，会抛出异常，由上层处理，不做自动重连。

---

## Q7：服务端在线编译

**面试官：** 你提到通过"临时目录隔离 + 动态依赖安装"来解决多用户并发编译冲突，能具体讲讲这个方案吗？比如每次编译请求来了，你的处理流程是怎样的？临时目录的生命周期是怎么管理的？动态依赖安装的性能问题你是怎么优化的？

**候选人：** 当编译请求来临时，再由创建一个 worker 去执行安装 NPM 包和编译、打包等操作，编译完成后的文件都会被放在一个临时目录下，打包完成后，再把打包后的文件回传给前端进行渲染。临时目录会在请求结束后自动删除。动态依赖安装使用 pnpm 安装，会自动缓存依赖。

---

## Q8：编译引擎细节

**面试官：** 第一，你说用 worker 去执行编译，这里的 worker 是指 Node.js 的 worker_threads，还是子进程 child_process，还是其他什么方式？选择它的原因是什么？第二，你提到支持 Webpack 和 esbuild 双引擎，什么场景下用 Webpack，什么场景下用 esbuild？用户可以选择，还是系统自动决定的？

**候选人：** 使用 child_process，首先故障隔离，CPU 密集型任务更适合用 child_process。用户有入口可以选择，我们默认是 webpack，这块没有特殊考虑，只是为了给用户多一个选择。

## Q9：Agent 架构演进

**面试官：** Workflow 编排和 Agent 自主决策，这两种架构在你们的业务场景里，核心区别体现在哪里？是什么业务痛点驱动你们做了这次架构升级？另外你设计的 Skill 抽象层，一个 Skill 的结构大概长什么样，它和原来 Workflow 的节点之间是怎么做映射和迁移的？

**候选人：** Workflow 和 Agent 自主决策的核心区别就是是否有 ReAct 的架构吧，也就是是否让模型对输出结果决策再执行的过程。之前的 workflow 模式，回复比较死板，同时后续迭代成本较大。之前我们的 Workflow，大概有三个子工作流，分别是数据处理，智能问答和营销热点推荐，迭代后，这三个子工作流分别有一个对应的 Skill。

---

## Q10：Skill 路由与准确率

**面试官：** 你说 Skill 和原来的子工作流是一一对应的，那 Agent 怎么决定调用哪个 Skill？是通过 function calling 的方式让模型自己选，还是你们有额外的路由逻辑？另外，有没有遇到过模型选错 Skill 的情况，你们是怎么优化准确率的？

**候选人：** Agent 会根据 skill.md 的 description 去选择执行一个或多个 skill。优化 Skill 的描述来提升准确率。

## Q11：职业规划

**面试官：** 你的求职意向写了"前端开发工程师/Agent 开发工程师"，你对自己未来的技术方向是怎么规划的？你觉得前端工程师在 AI 时代的核心竞争力是什么？如果到了新团队，你最希望做什么样的事情？

**候选人：** 首先我认为 AI 时代不应该限制自己的技术方向，AI 能够低成本的让你去了解一门技术。目前 AI 主要以执行为主，前端的核心竞争力目前可能还有些业务领域的决策以及具体方向上的决策。到了新团队，我希望能不限于前端领域，能够独立负责前后端多个领域的工作。

## 面试官综合反馈

**做得好的地方：** 技术选型有自己的判断依据，不是盲目跟风。比如 iframe 微前端的选型、child_process 做编译隔离、SSE 双重超时的设计，都能讲清楚背后的思考逻辑。对业务场景和技术方案的匹配度有比较好的把控。

**可以加强的方面：**

1. **回答展开度：** 比如聊 iframe 方案的痛点时，很快就说"其他问题可以接受"就带过了，但面试官其实是想看你对问题的认知深度。即使业务上可以接受，也可以主动聊聊你知道哪些优化手段、未来如果场景变了会怎么应对。

2. **Agent 回答深度：** 在 Skill 选择准确率优化这块只说了"优化描述"，但可以展开讲讲具体怎么优化的，比如有没有加 few-shot examples、有没有做意图分类的前置路由、有没有用评估数据来度量准确率等。这块是简历的亮点，值得多准备。

3. **自我介绍方式：** 建议不要直接念简历内容，可以用更自然的方式讲一条主线，比如"从 B 端业务打基础，到全栈独立做平台，再到 AI 产品落地"这样的成长脉络，会更有说服力。

4. **职业规划具体性：** 可以结合实际经历讲得更具体一些。比如已经有全栈和 Agent 的经验了，可以说清楚想往哪个具体方向深耕，而不是泛泛地说"不限于前端"。
